你现在扮演：资深量化策略工程师 + Rust 交易系统架构师 + Binance Futures API 专家。
我有一套可复用的 Rust 交易系统（已包含：交易对符号管理、精度/最小下单量/tick&step 处理、REST/WS 通道、下单/撤单、订单回报、仓位/余额查询等工具）。请基于它开发一个可长期运行的 SOLUSDC 永续合约策略模块：Hedge Mode（双向持仓）的“滚动固定数量网格 + 激进跟随 + 库存不足时少挂单（不硬凑K）”。

目标：能稳定运行数月甚至数年；收益来源是波动做市，不做任何收益保证；优先 Maker 挂单（post-only）。必须考虑：资金费率、标记价格、维持保证金/清算风险、下单精度、限速、断线恢复、订单一致性。

1) 交易标的与前置条件
============================================================
- 交易所：Binance Futures
- 交易对：SOLUSDC 永续
- 账户模式：必须为 Hedge Mode（双向持仓）。策略启动时检测，若不是则报错并停止（不自动切换）。

行情与风控参考：

2) 策略概念：固定挂单数量的“滚动网格”
============================================================
你要实现的行为：
- 在当前参考价 P_ref 上下各挂 K 档订单（K 可配置，范围 2..10）：
  - 下方 Buy Book：K 个买单价格梯子
  - 上方 Sell Book：K 个卖单价格梯子
- 网格间距为百分比几何间距（不是固定美元间距）：
  - 上方第 i 档：P_ref * (1 + grid_spacing_pct)^i
  - 下方第 i 档：P_ref * (1 - grid_spacing_pct)^i
  - 默认 grid_spacing_pct = 0.001（千分之一）
- 每档订单按固定名义下单：order_notional = 10 USDC（可配置）
  - qty = order_notional / price，按 step size 截断并满足 minQty/minNotional

成交后滚动（核心风格必须保留）：
- 当发生一次成交（fill，包含部分成交），执行一次滚动更新：
  - “新增 2 个订单 + 取消 1 个订单”，以维持固定的 K 档结构，并把 TP（止盈回补）挂回去。
- 同时增加“激进跟随”：当行情单边导致网格与现价出现空档，执行“补 1 撤 1”把梯子向现价方向滑动。

============================================================
3) 订单意图分类（必须清晰）
============================================================
策略内部必须将订单分为 4 类，并记录 purpose 字段（用于恢复、对账、风控）：

1) OPEN_LONG_BUY   : BUY  + positionSide=LONG   （开/加多）
2) CLOSE_LONG_SELL : SELL + positionSide=LONG   （平/减多；数量不得超过当前 LONG 持仓可用量）
3) OPEN_SHORT_SELL : SELL + positionSide=SHORT  （开/加空）
4) CLOSE_SHORT_BUY : BUY  + positionSide=SHORT  （平/减空；数量不得超过当前 SHORT 持仓可用量）

4) “库存不足时少挂单”的智能规则（你必须实现）
============================================================
问题：当 K=3，但实际 LONG 库存只够覆盖 1~2 个平仓卖单（CLOSE_LONG_SELL）时，若硬挂满 3 个平仓卖单会报错或导致逻辑错乱。

策略要求（按用户偏好）：数量不足时“少挂单”，不把平仓单硬拆到更小去凑 K。

实现规则：
A) 覆盖约束（硬性）
- 所有挂出的 CLOSE_LONG_SELL 的数量之和 <= 当前 LONG 持仓数量（可用量）
- 所有挂出的 CLOSE_SHORT_BUY 的数量之和 <= 当前 SHORT 持仓数量（可用量）
- 若不满足：立刻重建对应侧订单（见“重建流程”），只保留近端优先的平仓单，远端平仓单取消或缩量到可用范围（默认：取消远端，保留近端）。

B) 槽位分配（近端优先 + 不硬凑K）
- Sell Book（上方 K 档）优先放 CLOSE_LONG_SELL（有 LONG 库存才放），并且只放“能覆盖的档数”，不足的档数不强行拆分：
  - 例如 K=3，但 LONG 库存只够 2 单：只放 2 个 CLOSE_LONG_SELL（且放在离现价最近的两档）。
- Buy Book（下方 K 档）优先放 CLOSE_SHORT_BUY（有 SHORT 库存才放），不足同理只放可覆盖的档数，放在近端。

C) 不足的槽位怎么处理（必须可配置，默认激进做市）
- 参数：fill_remaining_slots_with_opens = true/false（默认 true）
- 若 true：平仓单不足的槽位，用开仓单补齐（仍保持每侧 K 档总订单数量）
  - Sell Book 的剩余槽位 -> OPEN_SHORT_SELL
  - Buy Book 的剩余槽位  -> OPEN_LONG_BUY
  - 注意：此行为受风控（净敞口/总名义/保证金）约束，风控不允许则留空。
- 若 false：平仓单不足时该侧订单数量就少于 K（只挂平仓单），不补开仓单。

用户已选择“数量不足时少挂单”，但这里的“少挂单”指“不硬凑平仓单数量”；是否用开仓单补齐由参数控制，默认 true（更符合你固定挂单数量与做市目标），同时必须严格风控。

5) 成交后的滚动更新规则（必须严格对称实现）
============================================================
输入：fill_event（包含：order_id、purpose、fill_qty、fill_price、positionSide、side、timestamp）

通用要求：
- 部分成交也要处理：按本次 fill_qty 生成对应 TP 单；剩余未成交部分保留在盘口（除非触发重建/风控撤单）。
- 每次滚动动作需遵守：频率限制 + 冷却时间（cooldown_ms），避免高频抖动。

规则 1：当 OPEN_LONG_BUY 成交（开多成交）
你必须做：
1) 新增 1 张止盈卖单（平多）：CLOSE_LONG_SELL
   - price = fill_price * (1 + grid_spacing_pct)
   - qty   = fill_qty
2) 新增 1 张“最远端补买单”（延伸买梯子）：OPEN_LONG_BUY
   - price = 当前 Buy Book 最远端价 * (1 - grid_spacing_pct)
     （若无，则按 P_ref*(1-grid)^K 再往下一档）
   - notional = order_notional
3) 为保持 Sell Book 总槽位不超过 K：
   - 取消 1 张“最远端卖单”（Sell Book 最高价那档）
   - 注意：取消的可能是 OPEN_SHORT_SELL 或 CLOSE_LONG_SELL（取决于当前槽位分配），但必须遵守“近端优先留平仓单”的原则：
     - 优先取消最远端的开仓卖单；若最远端是平仓卖单且平仓卖单数量已 > 可覆盖档数，则也可取消。

=> 这一步体现“新增2单 + 取消1单”，并维持结构。

规则 2：当 OPEN_SHORT_SELL 成交（开空成交）
对称操作：
1) 新增 1 张止盈买单（平空）：CLOSE_SHORT_BUY
   - price = fill_price * (1 - grid_spacing_pct)
   - qty   = fill_qty
2) 新增 1 张“最远端补卖单”（延伸卖梯子）：OPEN_SHORT_SELL
   - price = 当前 Sell Book 最远端价 * (1 + grid_spacing_pct)
3) 为保持 Buy Book 总槽位不超过 K：
   - 取消 1 张“最远端买单”（Buy Book 最低价那档），同样优先取消最远端开仓单，保留近端平仓单。

规则 3：当 CLOSE_LONG_SELL / CLOSE_SHORT_BUY 成交（平仓成交）
- 平仓成交代表风险下降：
  - 可按同样逻辑补回对侧开仓单以维持 K（可配置）
  - 或只做“结构重建”以确保近端有开仓梯子、平仓梯子符合库存覆盖
- 给出明确实现方案（默认）：
  - 平仓成交后触发一次“重建流程”（轻量）：
    - 更新持仓
    - 按槽位分配规则重新计算该侧应存在的平仓档数
    - 若因库存下降导致某些平仓单不再可覆盖，则取消远端平仓单，保留近端

6) 激进跟随（recenter / trailing）——解决单边行情不成交
============================================================
需求：当行情一直涨且 Long 没库存或买梯子离现价太远，原先提交的买单很久不会成交，导致做市停滞、仓位结构偏向一边。
你必须实现“按距离触发”的激进跟随，并保持固定槽位数量（或在风控下留空）。

参数：
- max_gap_steps = 1..2（默认 1，激进）
- follow_cooldown_ms = 300..1500（默认 800）
- max_follow_actions_per_minute（默认 30，防抖）

定义 gap_steps（用几何比例换算成格数）：
- gap_up = log(mid / best_buy_price) / log(1+grid)   （mid 高于最近买单多少格）
- gap_dn = log(best_sell_price / mid) / log(1+grid)  （最近卖单高于 mid 多少格）

触发与动作（逐步迁移，不全撤全挂）：
A) 上涨时（mid 远离最近买单）
若 gap_up > max_gap_steps 且满足 follow_cooldown：
- 在 Buy Book 近端补 1 张新 OPEN_LONG_BUY（或在风控允许且需要时补 CLOSE_SHORT_BUY）：
  - 推荐价格（更稳不追太狠）：new_buy = best_buy_price * (1 + grid)
  - 若要更激进：new_buy = mid * (1 - grid)
- 同时取消 Buy Book 最远端 1 张订单（最低价那档）
- 该动作称为“补1撤1”，重复执行可把买梯子整体向上滑动

B) 下跌时（mid 远离最近卖单）
若 gap_dn > max_gap_steps：
- 在 Sell Book 近端补 1 张新 OPEN_SHORT_SELL（或风控允许时补 CLOSE_LONG_SELL）：
  - new_sell = best_sell_price * (1 - grid)  或  new_sell = mid * (1 + grid)
- 取消 Sell Book 最远端 1 张订单（最高价那档）

注意：激进跟随必须受风控阀约束（见下一节）；风控不允许开仓则只能做“撤远端单/保留减仓单”。

============================================================
7) 风控阀（必须实现，优先级 > 继续挂单）
============================================================
你必须实现至少以下风控，并明确触发->动作：

1) max_net_notional（净敞口上限）
- net_notional = long_notional - short_notional（用 mark price）
- 若 |net_notional| 超过阈值：
  - 禁止导致净敞口进一步扩大的 OPEN_* 补单与激进跟随补单
  - 仅允许减少净敞口的平仓单继续工作
  - 可选：进入 defensive_mode

2) max_total_notional（总名义上限）
- total_notional = abs(long_notional) + abs(short_notional)
- 若超过阈值：
  - 禁止任何增加 total_notional 的开仓单（OPEN_*）
  - 允许平仓单（CLOSE_*）与撤远端单

3) margin safety buffer（保证金安全垫）
- 监控 equity、maintenance margin、margin ratio、liquidation risk（以交易所返回为准）
- 若低于安全阈值：
  - 立即取消所有 OPEN_* 挂单
  - 只保留 CLOSE_*（且近端优先）
  - 必要时可触发主动减仓（可配置，默认关闭）

4) funding control（资金费率控制）
- 定期读取 funding rate 与结算时间
- 估算 expected_funding_cost（按 total_notional 与费率）
- 若预计资金费吞噬网格毛利或显著不利：
  - 降低开仓补单频率 / 暂停跨结算窗口开仓（只留平仓与撤单）

5) kill switch（熔断）
- 触发：连续下单失败、WS断连超时、状态对账不一致无法自愈、权益回撤超阈值等
- 动作：撤掉所有挂单、停止开仓、输出报警与恢复步骤

============================================================
8) 错误处理与自愈（必须有）
============================================================
- post-only 拒单（会吃单）：价格外移 1 tick 重试，最多 retry_n 次；失败则放弃该档并触发一次轻量重建。
- 精度/minNotional/minQty 拒单：调整 qty/price 到合法值；若仍不合法则跳过该档并记录告警。
- 平仓数量不足：立刻取消该平仓单，触发“槽位重建”（近端优先），绝不让策略反手。
- 订单/仓位对账：定时 REST 拉 openOrders + positions 与本地账本一致性校验；发现偏差自动重建（撤掉不认识的订单或重新登记）。
