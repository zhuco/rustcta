# Aster / MEXC / KuCoin Futures / Bybit 合约功能补全开发文档

日期：2026-06-12

## 1. 目标与边界

本文档用于指导 `aster`、`mexc`、`kucoinfutures`、`bybit` 四个交易所的 USDT 本位永续合约能力补全，目标是满足当前两类策略的实盘前置接口要求：

- 跨交易所合约套利策略：`strategies/cross-exchange-arbitrage` 与 `crates/rustcta-cross-arb-live-runner`。
- 资金费率套利 / funding spread expansion maker 策略：`strategies/funding-arbitrage` 与 `docs/funding_spread_expansion_maker_strategy_design_zh.md`。

边界：

- 本轮只写开发文档，不修改交易执行代码。
- `kucoin` 是现货适配器，合约交易必须走 `kucoinfutures`。不要把 KuCoin 现货能力误报为合约能力。
- 本文只覆盖 USDT 线性永续合约。交割合约、币本位合约、期权、现货保证金不纳入本轮策略闭环。
- 能力声明必须以适配器真实实现和测试为准。没有请求构造、响应解析、错误处理和 fixture 的能力，不得只因为官方支持就打开 capability。

## 2. 策略需要的公共接口

### 2.1 跨交易所合约套利最小闭环

运行跨交易所双边合约套利时，每个参与交易所至少需要以下能力：

| 类别 | 必需接口 / 语义 | 用途 |
| --- | --- | --- |
| 合约元数据 | `get_symbol_rules` | 获取 tick size、数量步进、最小数量、最小名义价值、合约状态、合约乘数。 |
| 盘口快照 | `get_order_book` | 启动时构建初始盘口，WebSocket 断线或序列断档后重建。 |
| 公共 WebSocket | order book L1 / depth | 高频触发价差判断，要求能识别序列缺口并回落 REST snapshot。 |
| 资金费率 | `get_funding_rates` | 计算资金费率成本、跨所持仓方向偏好和资金费率套利信号。 |
| 手续费 | `get_fees` | 计算 taker / maker 成本与价差阈值。 |
| 账户 | `get_balances`、`get_positions` | 入场前额度检查、仓位对齐、风险敞口检查。 |
| 下单 | `place_order` | 支持限价、市场、IOC/FOK、reduce-only、client order id。 |
| 撤单 | `cancel_order`、`cancel_all_orders` | 超时撤单、异常清理、策略停机保护。 |
| 订单回读 | `query_order` 或 `get_open_orders` | 私有 WS 不可用时确认订单状态。 |
| 成交回读 | `get_recent_fills` 或 `query_order` 成交字段 | 确认成交数量、均价、手续费，驱动对冲腿修复。 |
| 私有 WebSocket | order / execution / position / wallet | 推荐能力；缺失时必须明确走 REST readback fallback。 |
| 账户控制 | leverage、position mode | 实盘启动前统一杠杆与持仓模式。unsupported 时必须在配置和 admission check 中显式阻断或人工预配置。 |

`crates/rustcta-cross-arb-live-runner/src/lib.rs` 当前 admission check 已硬性要求：

- `private_rest`。
- `positions`。
- `place_order`。
- `cancel_order`。
- `query_order` 或 `open_orders`。
- `recent_fills` 或 `query_order`。
- `reduce_only`。
- IOC time-in-force。
- IOC order type。
- Market order type。
- Slippage capture 模式额外要求 post-only maker 能力。

### 2.2 资金费率套利 / funding spread expansion maker 需求

资金费率类策略除 2.1 的交易闭环外，还需要更完整的扫描和持仓数据：

| 类别 | 必需接口 / 数据 | 说明 |
| --- | --- | --- |
| funding snapshot | 当前资金费率、预测资金费率、下一次结算时间、mark price | `FundingRateSnapshot` 必须尽量填满；缺 `next_funding_time` 时不能进入临近结算策略。 |
| mark / index price | mark price 必需，index price 推荐 | 用于基差和强平风险估算。当前共享模型缺 index 时可先放在扩展字段或策略侧补充。 |
| 盘口深度 | top of book 与可配置深度 | maker/taker 或 dual maker 模式需要估算排队、滑点和撤单风险。 |
| 费率 | maker / taker fee | 资金费率净收益必须扣除手续费和预计滑点。 |
| 仓位 | long / short / one-way 模式映射 | 必须能区分双向持仓模式下的 long/short leg。 |
| 挂单能力 | post-only、cancel、open orders | maker 扩展策略必须有稳定撤单和挂单查询。 |
| 成交确认 | private stream 或 REST readback | 资金费率临界窗口不能假设提交成功就是成交。 |
| open interest | 推荐 | 用于筛选流动性和拥挤度；不应阻塞 P0 交易闭环。 |
| funding settlement ledger | P2 | 资金费率结算后做 PnL 对账，当前共享接口未统一。 |

## 3. 当前适配器状态与主要缺口

### 3.1 总览矩阵

| 交易所 | 适配器 | 当前可用基础 | 策略关键缺口 | 优先级 |
| --- | --- | --- | --- | --- |
| Aster | `aster` | Futures V3 公私 REST、下单/撤单/查询/仓位/余额/费率、公共与私有 WS 框架、Binance 风格深度序列文档。 | 共享 `get_funding_rates` 未闭环；杠杆与持仓模式官方可用但适配器未暴露 account control；私有 WS 事件归一化和 ready gate 需要验证；官方支持 amend/batch 但未实现；order book capability 仍偏保守。 | P0 |
| MEXC | `mexc` | 合约 public/private REST、资金费率、仓位、余额、费率、下单/撤单/查询/成交、杠杆；私有 WS 以 REST fallback 声明。 | 需用当前官方 futures 文档重新核对路径和字段，尤其下单路径、reduce-only、position mode、side/type/openType 映射；私有 WS 未接入；合约 WS 增量盘口频率和序列语义需 canary。 | P0 |
| KuCoin Futures | `kucoinfutures` | Classic futures public/private REST、funding、symbol rules、order book snapshot、订单/成交/仓位/余额/费率、public/private bullet token WS。 | 杠杆、保证金模式、持仓模式未暴露；classic 与 UTA API 边界需固定，不能静默混用；order book capability 偏保守；open interest 扫描缺口；private WS token 续期和重订阅需要回归测试。 | P0/P1 |
| Bybit | `bybit` | V5 REST 覆盖度最高：funding、symbol rules、order book、订单/批量/改单/撤单/成交/仓位/余额/费率、杠杆、持仓模式；公共/私有 WS 文档映射完整。 | 需要把 public WS orderbook.1/50 运行链路和序列恢复纳入策略 profile；私有 WS order/execution/position/wallet parser 与 readiness 要实测；open interest、mark/index 扫描数据需补齐；close_position/countdown cancel 仍不是共享能力。 | P0/P1 |

### 3.2 关键判断

- Bybit 是四个交易所中最接近可实盘闭环的适配器，应作为 capability gate 和测试模板。
- Aster 的官方 Futures V3 接口与 Binance Futures 风格接近，最紧急缺口是 shared funding API、杠杆/持仓模式和私有 WS 事件闭环。
- MEXC 当前代码已有较多合约功能，但官方文档路径有版本漂移风险；实现前必须先做 endpoint audit 和 fixture 校准。
- KuCoin 必须明确 `kucoinfutures` classic futures 与 UTA futures 的边界。短期不建议把 UTA 端点塞进 classic 适配器导致行为混杂。

## 4. 分阶段实施计划

### P0：让两个策略能安全 admission

1. 补齐 Aster `get_funding_rates`
   - 对接 `GET /fapi/v3/premiumIndex` 与 `GET /fapi/v3/fundingRate`。
   - 输出 `funding_rate`、`predicted_funding_rate`、`funding_time`、`next_funding_time`、`mark_price`。
   - 若官方只返回当前值和下一次结算时间，`predicted_funding_rate` 可以为 `None`，不得伪造。

2. MEXC 官方路径刷新
   - 对照当前官方 futures 文档核对下单、撤单、查询、open orders、fills、余额、仓位、费率、杠杆接口。
   - 特别检查当前代码使用的 `/api/v1/private/order/submit` 是否应迁移或兼容 `/api/v1/private/order/create`。
   - 为 open long、open short、close long reduce-only、close short reduce-only 建 fixture，确保 side/type/openType/positionMode/reduceOnly 映射无歧义。

3. 统一 capability gate
   - 为 `aster`、`mexc`、`kucoinfutures`、`bybit` 写策略级能力测试。
   - 测试应覆盖 `trade_capability_requirements` 中的硬性要求。
   - funding 策略还要检查 funding snapshot 是否包含 `next_funding_time` 或明确声明不可用于临近结算窗口。

4. 公共 WebSocket 盘口 profile
   - Aster：`bookTicker`、`depth5/10/20@100ms`，以及 diff depth + REST snapshot replay。
   - MEXC：先使用 `sub.depth.full` 保守 profile；`sub.depth` 增量 profile 需要 canary 验证序列与延迟。
   - KuCoin Futures：classic `level2Depth5/50` 或 level2 增量，必须有 sequence gap resync。
   - Bybit：linear `orderbook.1` 做最快 L1，`orderbook.50` 做深度滑点估计；使用 `u`/`seq` 做缺口判断。

5. 私有确认链路
   - 优先使用 private WebSocket 归一化 order/execution/position/wallet。
   - 私有 WS 未完成的交易所必须在 capability v2 中标为 REST fallback，并在 live 配置中要求 REST readback confirmation。
   - 任何 REST readback 失败都不能当作已成交处理。

6. 账户控制
   - Bybit：保持 leverage、position mode 可用，并补测试。
   - Aster：实现 leverage、position side dual 的 account control。
   - MEXC：保留 leverage；position mode 只有官方接口和映射验证后才打开。
   - KuCoin Futures：未完成官方核验前保持 leverage / position mode unsupported，要求人工预配置或策略 admission 阻断。

### P1：提升 maker 与高频稳定性

1. 私有 WS 事件归一化
   - Aster：listenKey 创建、续期、删除；解析 account/order trade update。
   - MEXC：若官方仍提供合约私有 WS，补 auth、订阅、order/fill/balance/position parser；否则保持 REST fallback。
   - KuCoin Futures：强化 bullet private token 获取、续期、重连后重订阅、REST resync。
   - Bybit：完善 private `order`、`execution`、`position`、`wallet` parser 和 ready gate。

2. maker 能力
   - Aster：实现官方 `PUT /fapi/v3/order` 改单、`POST/DELETE /fapi/v3/batchOrders` 批量接口。
   - MEXC：确认是否有原生改单/批量接口；没有则维持 cancel+place 组合语义并清楚标注非原子。
   - KuCoin Futures：classic futures 不支持的能力不打开；如 UTA 支持，另建 UTA profile。
   - Bybit：已有 amend 和 batch，补充策略侧 fixture 和异常码处理。

3. 扫描增强
   - 补 open interest、24h turnover、mark/index price 的 parser。
   - 这些数据先作为 scanner enrichment，不阻塞 P0 下单闭环。

### P2：统一高级账户和对账能力

1. 设计共享 margin mode mutation request。
2. 设计通用 close position / flatten position 接口，或明确由策略以 reduce-only IOC 组合实现。
3. 设计 countdown cancel all / dead-man switch 抽象。
4. 设计 funding settlement ledger 接口，用于资金费率到账后 PnL 对账。
5. 单独规划 KuCoin UTA futures 适配器或 profile，不与 classic futures 静默混用。

## 5. 单交易所任务清单

### 5.1 Aster

P0-A1：资金费率共享接口

- 文件范围：`crates/rustcta-exchange-gateway/src/adapters/aster/`。
- 端点：`GET /fapi/v3/premiumIndex`、`GET /fapi/v3/fundingRate`。
- 输出：`FundingRateSnapshot`。
- 测试：
  - 单 symbol。
  - 多 symbol 或全量 symbol。
  - 缺历史 funding 时只返回当前 premium snapshot。
  - timestamp 毫秒转换为 UTC。

P0-A2：账户控制

- 端点：`POST /fapi/v3/leverage`。
- 端点：`POST /fapi/v3/positionSide/dual`、`GET /fapi/v3/positionSide/dual`。
- 共享接口：`PerpAccountControlProvider`。
- 注意：只有私有 REST 凭证可用时才声明支持。

P0-A3：公共 WebSocket 盘口

- 支持 `bookTicker` 快速 L1。
- 支持 `depth5@100ms`、`depth10@100ms`、`depth20@100ms`。
- 支持 diff depth 的 `U/u/pu` 连续性校验和 REST snapshot replay。
- capability 从 `snapshot_only` 升级前必须有 parser fixture 和序列断档测试。

P0-A4：私有 WebSocket

- listenKey 创建、续期、删除。
- 解析订单成交、账户更新、仓位更新。
- 启动时必须有 ready gate，断线后必须 REST resync。

P1-A5：高级订单

- 官方存在改单和批量接口，可作为 maker 策略优化项。
- 批量接口要返回逐单结果，不能把部分失败当整体成功。

### 5.2 MEXC

P0-M1：官方 futures endpoint audit

- 核对 REST base：`https://contract.mexc.com`。
- 核对下单接口：当前代码路径与官方 `/api/v1/private/order/create` 差异。
- 核对取消、批量取消、查询、open orders、fills、余额、仓位、手续费、杠杆路径。
- 所有路径变更都要更新 endpoint mapping、request spec fixture、parser fixture。

P0-M2：合约下单语义测试

- open long。
- open short。
- close long reduce-only。
- close short reduce-only。
- one-way / hedge mode 的 position side 映射。
- post-only、IOC、FOK、market、limit 的 type 映射。

P0-M3：position mode

- 如果当前官方 API 支持修改 position mode，则实现 account control 并补 readback。
- 如果只能在网页或账户级预配置，则保持 unsupported，并在 strategy admission 中输出明确错误。

P0-M4：公共 WebSocket 盘口 profile

- 保守 profile：`sub.depth.full`。
- 候选高频 profile：`sub.depth` 增量。
- 需要 canary 验证更新频率、序列字段、断档恢复、REST snapshot 对齐。

P1-M5：私有 WebSocket

- 若官方支持合约私有 WS，补 auth、订阅、order/fill/position/balance parser。
- 若不可用，继续把 private stream capability 标为 REST fallback，并强化 readback 节流和超时。

### 5.3 KuCoin Futures

P0-K1：适配器边界

- `kucoin` 只用于现货。
- `kucoinfutures` 用于 classic futures 合约。
- UTA futures 如需支持，应单独设计 profile 或新适配路径，不要覆盖 classic futures 默认行为。

P0-K2：盘口序列与重建

- classic futures level2 增量和 snapshot 必须有 sequence 对齐测试。
- 如果只保证 snapshot，则 capability 保持保守，不得声明 strict delta。

P0-K3：账户控制核验

- 当前 leverage、margin mode、position mode 不应默认声明。
- 若 classic futures 只在下单参数中携带 leverage，则不等价于账户级 `set_leverage`。
- 实现前需要官方文档、request spec、readback 确认三件套。

P0-K4：private bullet WS 稳定性

- token 获取。
- token 续期。
- 断线重连。
- 重订阅。
- REST resync orders / fills / balances / positions。

P1-K5：UTA futures 规划

- UTA 下单、资金费率、仓位、杠杆、私有 token 端点与 classic futures 不同。
- 先做官方核验文档和 fixture，再决定是否新增 adapter profile。

### 5.4 Bybit

P0-B1：公共 WebSocket 盘口运行链路

- linear `orderbook.1`：最快 L1 触发。
- linear `orderbook.50`：滑点和深度估计。
- 使用 `u`、`seq` 判断缺口；断档后 REST snapshot 重建。
- capability 升级必须跟 parser fixture 一起提交。

P0-B2：私有 WebSocket

- auth。
- `order`、`execution`、`position`、`wallet` 订阅。
- ready gate。
- 断线后 REST resync。
- 与 REST `query_order`、`get_recent_fills` 的成交去重。

P0-B3：扫描数据

- open interest。
- mark price / index price。
- funding history 与 next funding time。
- 24h volume / turnover。

P1-B4：高级保护

- close position 可以先由策略生成 reduce-only IOC，不必急于新增共享接口。
- countdown cancel all / dead-man switch 若官方可用，应先抽象共享 request，再实现 Bybit。

## 6. 测试与验收标准

### 6.1 每个交易所最少测试

- capability snapshot 测试。
- request spec 测试：签名、method、path、query/body、headers。
- parser fixture 测试：成功响应、空响应、错误响应。
- order semantic 测试：market、limit、IOC、FOK、post-only、reduce-only、client order id。
- position semantic 测试：one-way、hedge long、hedge short。
- funding parser 测试：funding rate、predicted funding rate、funding time、next funding time、mark price。
- order book WS 测试：snapshot、delta、sequence gap、resync。
- private stream 测试：order ack、partial fill、full fill、cancel、reject、position update、wallet update。

### 6.2 策略级验收

- `cargo check -p rustcta-exchange-gateway`。
- `cargo test -p rustcta-exchange-gateway --all-features`。
- `cargo test -p rustcta-cross-arb-live-runner --all-features`。
- 能力矩阵中四个交易所对 `trade_capability_requirements` 的结果符合预期。
- funding 策略 scanner 能读取四个交易所同一 symbol 的 funding snapshot、symbol rules、fee、order book。
- 无真实下单前，先跑只读 canary：symbol rules、funding、fee、balance、position、public WS order book、private WS auth。
- 小额实盘前必须启用 readback confirmation 和 reduce-only repair。

## 7. 安全规则

- 凭证缺失时不得声明 private REST / private WS / account control。
- 官方支持但本地未实现时，不得打开 capability。
- REST readback 超时、订单状态 unknown、私有 WS 断线时，策略必须进入保护逻辑，而不是继续加仓。
- reduce-only 修复订单必须走独立限额和 client order id 前缀。
- batch order 只在逐单结果解析完整后才能用于策略；否则维持单笔下单。
- position mode / leverage 修改必须在无未成交挂单或策略启动前执行。
- KuCoin classic futures 与 UTA futures 不得混用 symbol、position、order id 语义。
