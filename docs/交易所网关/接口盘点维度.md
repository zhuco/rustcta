# 交易所接口盘点维度

状态日期：2026-06-08

这份文档用于给每个交易所补接口文档。不要只按“币安现货”照抄；先按产品线判断这个交易所到底有什么业务，再填能力矩阵。没有现货、没有合约、没有私有接口的地方，统一写 `交易所不支持`，不要留空。

## 状态口径

每个能力只允许使用下面这些状态：

| 状态 | 含义 |
| --- | --- |
| `原生支持` | 交易所有官方接口，语义可以无损映射到网关接口。 |
| `组合支持` | 需要多个官方接口组合完成，例如先查 open orders 再逐个 cancel。 |
| `REST 兜底` | WebSocket 或批量能力不足，但可以用 REST readback/reconciliation 保证状态。 |
| `仅 WS` | 只适合流式订阅，不适合普通 REST 查询。 |
| `不支持` | 交易所没有兼容接口，或暂未验证，网关必须显式返回 Unsupported。 |
| `交易所不支持` | 交易所没有这条产品线或官方没有该能力，例如没有合约、没有现货、没有期权。 |

## 一级维度

| 维度 | 必填内容 | 目的 |
| --- | --- | --- |
| 交易所身份 | adapter id、官网、官方文档、区域限制、主网/测试网 host | 确认接的是哪套 API，避免同名国际站/美国站/欧盟站混用。 |
| 产品线 | Spot、Margin、USDT 永续、币本位、交割合约、Option、Broker、链上永续 | 先判断 `MarketType`，没有的产品线填 `交易所不支持`。 |
| 账户模型 | 统一账户/经典账户、子账户、组合保证金、持仓模式、资产币种 | 影响余额、仓位、下单参数和风控解释。 |
| 公共 REST | symbol rules、order book snapshot、ticker、trades、candles、funding、open interest | 扫描、回放、订单精度和盘口重建的基础。 |
| 私有读取 | balances、positions、fees、query order、open orders、recent fills、order history | 下单前检查、下单后对账、异常恢复。 |
| 私有写入 | place、quote market、cancel、cancel-all、amend、batch place/cancel、order list | 交易执行面的核心能力。 |
| 公共 WS | order book delta、trades、ticker、candles、订阅/退订、心跳、重连、推流间隔、档位深度 | 跨所套利的核心行情来源，必须细分延迟和订单簿能力。 |
| 私有 WS | orders、fills、balances、positions、account、登录、续期、断线恢复 | 订单生命周期和账户事件。 |
| 衍生品扩展 | leverage、margin mode、position mode、reduce-only、close-position、funding | 永续/期货不能用现货维度替代。 |
| 约束与风控 | rate limit、权限 scope、client order id、精度、最小名义额、限价类型 | 决定 adapter 前置校验和失败分类。 |
| 对账与恢复 | timeout 后 query、open orders、fills、REST/WS 补偿、未知订单策略 | 防止重复下单和状态漂移。 |
| 测试资产 | endpoint mapping、request spec、signing vector、parser fixture、WS fixture | 每个能力必须能离线验证，live 只作为最后验证。 |

## 产品线矩阵

每个交易所先填这一张表，决定后续章节是否需要展开。

| 产品线 | MarketType | 是否存在 | 网关目标 |
| --- | --- | --- | --- |
| 现货 | `Spot` | `原生支持` 或 `交易所不支持` | symbol rules、盘口、余额、费率、订单生命周期。大多数平台应有现货；少数没有就明确写 `交易所不支持`。 |
| 杠杆现货 | `Margin` | `原生支持` 或 `交易所不支持` | 只在账户和借贷语义能清楚隔离时接入。 |
| USDT/USDC 永续 | `Perpetual` | `原生支持` 或 `交易所不支持` | 很多现货平台没有合约；没有就明确写 `交易所不支持`。 |
| 币本位期货/永续 | `Futures`/`Perpetual` | `原生支持` 或 `交易所不支持` | 合约面值、张数换算、币本位余额和仓位。 |
| 期权 | `Option` | `原生支持` 或 `交易所不支持` | 先做只读，交易能力必须单独审核 Greeks/行权语义。 |
| Broker/区域站 | 按实际产品 | `原生支持` 或 `交易所不支持` | 标清楚不是普通 CEX API，权限和账户模型单独说明。 |
| 链上永续/DEX | `Perpetual` 或按实际产品 | `原生支持` 或 `交易所不支持` | 钱包签名、链上状态、撮合和索引器边界必须单列。 |

## 标准接口矩阵

每个交易所文档至少要覆盖这些操作：

| 分类 | 网关操作 | 常见 Binance 基准 | 其他产品线如何填 |
| --- | --- | --- | --- |
| 能力 | `get_capabilities` | 返回产品线、REST/WS、订单能力、批量能力 | 按实际产品线填；没有 Spot 就写 `交易所不支持现货`。 |
| 公共规则 | `get_symbol_rules` | `/api/v3/exchangeInfo` | 永续用合约信息接口；DEX 用市场元数据；没有稳定接口写不支持。 |
| 公共盘口 | `get_order_book` | `/api/v3/depth` | 永续/DEX 用对应 order book 或链上/索引器来源。 |
| 账户余额 | `get_balances` | `/api/v3/account` | 永续用 futures balance；链上钱包不等于网关余额，需单独说明。 |
| 仓位 | `get_positions` | 现货没有合约仓位 | 永续/期货必须填；只有现货的交易所写 `交易所不支持合约仓位`。 |
| 费率 | `get_fees` | commission/trade fee | 没有私有费率接口时说明是否用配置兜底。 |
| 下单 | `place_order` | market/limit/post-only/IOC/FOK | 按产品线填支持的 order type、TIF、reduce-only。 |
| 市价按金额 | `place_quote_market_order` | Spot quoteOrderQty | 永续通常写 `交易所不支持按金额合约下单`，除非官方支持金额下单。 |
| 撤单 | `cancel_order` | orderId/origClientOrderId | 必须说明支持哪些 id。 |
| 全撤 | `cancel_all_orders` | symbol-scoped openOrders | 区分全账户、按 symbol、按产品线。 |
| 改单 | `amend_order` | keep-priority amend | 没有原生修改时不要用撤单重下冒充原生能力。 |
| 组合单 | `place_order_list` | OCO/OTO | 官方没有 OCO/OTO 等组合单时写 `交易所不支持组合单`。 |
| 批量下单 | `batch_place_orders` | Spot 当前未实现，USD-M 可对照 | 没有原生批量时可评估组合支持。 |
| 批量撤单 | `batch_cancel_orders` | 按 id 列表 | 需要说明原子性和部分失败。 |
| 查单 | `query_order` | orderId/origClientOrderId | 对账必填；没有则需要替代策略。 |
| 当前挂单 | `get_open_orders` | symbol optional | 对账和全撤组合能力依赖它。 |
| 成交 | `get_recent_fills` | myTrades | 必须说明 from id、时间窗口、limit。 |
| 公共订阅 | `subscribe_public_stream` | depth/trade/ticker/kline | 写清楚订阅方式、推流间隔、档位、心跳、序列号、重建快照。 |
| 私有订阅 | `subscribe_private_stream` | listen key/user data stream | 写清楚 auth、续期、断线后的 REST reconciliation。 |

## WebSocket 行情必须盘点的细项

跨所套利不能只知道“有没有 WebSocket”。每个交易所必须单独填下面这些信息，尤其是订单簿：

| 维度 | 必填内容 | 为什么重要 |
| --- | --- | --- |
| 订阅方式 | URL path stream、JSON subscribe、combined stream、multiplex、多 symbol 单连接、单 symbol 单连接 | 决定连接数、订阅管理、退订方式和故障隔离。 |
| 订单簿类型 | snapshot、delta、partial depth、best bid/ask、bookTicker、L1、L5、L10、L20、L50、L100 等 | 跨所套利最少需要稳定 best bid/ask；深度撮合需要更多档。 |
| 推流间隔 | 例如 10ms、20ms、50ms、100ms、250ms、1000ms，或交易所只写 real-time；必须标官方来源和核对日期 | 决定延迟模型和机会过期时间。不能只写“实时”。 |
| 档位深度 | 1 档、5 档、10 档、20 档、50 档、100 档、全量增量 | 决定可交易数量估算，1 档适合报价，深档适合大额成交模拟。 |
| 是否可选 speed/depth | 订阅参数是否能指定 `@100ms`、`level=1`、`depth=5`、`freq=10ms` 等 | 同一交易所可能既有慢速深档，也有高速 1 档。 |
| 事件时间字段 | exchange event time、transaction time、local receive time | 计算跨所延迟和 stale book。 |
| 序列字段 | first update id、last update id、prev update id、sequence、checksum | 判断丢包、乱序和是否需要重建。 |
| 快照重建 | REST snapshot endpoint、snapshot depth、delta buffer 规则 | delta book 没有重建规则就不能用于真实套利。 |
| 校验方式 | checksum、top-N checksum、无 checksum | 没有 checksum 时要提高重建频率或降级使用。 |
| 心跳 | server ping、client ping、pong、超时阈值 | 影响连接保活和断线判断。 |
| 限制 | 每连接订阅数、每秒订阅次数、连接数、消息频率限制 | 决定多交易所多 symbol 的部署规模。 |
| 退订 | 是否支持 unsubscribe、是否需要重连才能变更 symbol | 影响动态 symbol 池。 |

推荐把订单簿能力再细分成下面这张表：

| 行情能力 | 状态 | 推流间隔 | 档位 | 订阅示例 | 官方来源/核对日期 | 重建/备注 |
| --- | --- | --- | --- | --- | --- | --- |
| L1 best bid/ask | 待填写 | 待填写 | 1 | 待填写 | 待填写 | 适合最快报价监控。 |
| Partial depth | 待填写 | 待填写 | 5/10/20/50 等 | 待填写 | 待填写 | 适合轻量深度估算。 |
| Full depth delta | 待填写 | 待填写 | 全量增量 | 待填写 | 待填写 | 必须有 snapshot + sequence 规则。 |
| Trade stream | 待填写 | 待填写 | 不涉及 | 待填写 | 待填写 | 用于成交冲击和活跃度判断。 |
| Ticker/mini ticker | 待填写 | 待填写 | 不涉及 | 待填写 | 待填写 | 只能辅助，不能替代盘口。 |

## 文档完成标准

一份交易所接口文档算“可执行”，至少满足：

- 每个产品线都明确为 `原生支持`、`组合支持`、`REST 兜底`、`仅 WS`、`不支持` 或 `交易所不支持`。
- 每个网关操作都有 endpoint、认证方式、权限 scope、rate limit、分页或批量限制。
- 每个写操作都说明 client order id、幂等、超时后对账策略。
- 每个 WebSocket 通道都说明订阅方式、推流间隔、档位、心跳、重连、是否需要重新订阅、是否需要 REST snapshot。
- 每个 unsupported 都写原因，不使用“待确认”作为最终状态。
- 至少有 endpoint mapping、request spec、parser fixture、signing vector 或 WS fixture 中的一类离线证据。
