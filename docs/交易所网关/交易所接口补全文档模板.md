## REST 接口矩阵

官方存在但当前 adapter 未接入稳定 runtime 的 REST 操作，在矩阵里写 `项目未实现`，并在 `endpoint_mapping.yaml` 的 `unsupported:` 对应 operation 上标 `status: project_unimplemented`；当前 profile 不应接入的能力只写 `不支持` 和原因。
产品线项目缺口以矩阵 `official_gap_check=项目未实现 *_product` 为准；`contract_project=不支持` 只表示标准 futures/perpetual/options 不支持，仍可与 Margin、Leveraged、第三方 Perps、Event Contracts 等独立产品线的 `项目未实现` 共存。Margin/Leveraged/区域特殊产品必须有 `status: project_unimplemented`、`boundary: project_unimplemented_product_line`、source fixture classification 和 required audit gaps。Spot CEX adapter 缺 Futures/Perpetual/Derivatives/Margin/CFD 时，至少列出 contract metadata、合约盘口、position/risk、private order lifecycle、auth/scope guard 和 unsafe-to-promote 条件；只存在现货 runtime、产品页、合约费率来源或离线 fee request-spec，不能写成完整合约 runtime。同一交易所有多个 futures profile 时也必须逐 profile 审计：已有 USDT-M/USDT-FUTURES runtime，不等于 USDC-M、COIN-M、Delivery、Margin 或 Portfolio/UTA profile 已接入；必须有 productType/settlement coin guard、metadata/book、account/position/risk、private order lifecycle 和 post-write reconciliation 后才能提升。
衍生品/链上 venue 缺 Spot/OpenBook 时，不要只写一句 `交易所不支持现货`：在 `unsupported:` 中保留 `spot_product status: project_unimplemented`，并写清 `current_scope`、`runtime_gap`、`required_endpoints` 或 `required_sources`。例如 perp/options adapter 缺 Spot，应列 spot products/markets、spot order book、account/balance、order/open-order/fills、链上 book/event queue、wallet tx builder 或 indexer reconciliation 等后续项。
`get_balances` 是账户状态 readback，不强制等同 API key 认证，也不要因为认证方式不同就判 `不支持`：中心化交易所通常走 signed private REST；DEX/链上 profile 可以走 wallet/address、account id、account object、node 或 indexer 读取，部分场景不需要交易所认证但仍是有效余额来源。只有当前官方/profile 明确不承接账户余额 runtime 时才写 `不支持`；未接入时写 `项目未实现` 或离线 source 边界，并写清地址作用域、资产精度、重组/延迟和对账策略。
`get_positions` 同样要按 profile 区分来源：中心化合约通常走 signed position endpoint 或 private WS；DEX/链上 perpetual 可以走 wallet、contract、account object、SDK 或 indexer readback。官方存在读取路径但当前未接入时写 `项目未实现`，不要误写成交易所不支持。
`get_fees` 要区分账户有效费率 API、官方费率表/配置型 fee source、协议费率参数和成交回报中的手续费金额：官方有 maker/taker fee endpoint、VIP tier table、协议费率参数或明确零费率模型但当前未接入统一 `FeeRateSnapshot` 时，若只有文字证据写 `项目未实现`；若标准 `get_fees` operation 已有 request-spec/signing/response fixture、fee table fixture 或 source-boundary fixture 但 runtime/parser 未启用，矩阵必须识别为 `离线`。默认零费率占位、成交 fee amount、未验证 fallback 不能写成稳定费率 runtime，必须标明 source、适用产品线、VIP/account tier、配置覆盖策略、scope/version guard 和剩余 parser/runtime promotion。
核心交易待办按 operation 记录，不按 adapter 一刀切：如果 place/query/open/fills 已运行，但 cancel 或 query 仍是 `项目未实现`，任务行必须收窄到剩余 operation；如果 request-spec 已补但 runtime/parser/dry-run 未启用，写 `离线边界已补，shared runtime 未启用`。
高级订单能力也按同一规则处理：官方有 native batch place/cancel、amend/cancel-replace、bulk order 或链上 replace/cancel tx 线索，但当前 adapter 缺签名、request spec、parser、dry-run guard 或共享 trait 接入时，写 `项目未实现` 并在 `unsupported:` 对应 operation 标 `status: project_unimplemented`；只有官方无可无损映射的 OCO/OTO/order-list、in-place amend 或 batch 语义时，才写 `交易所不支持`。
高级订单 P4 待办表只保留矩阵状态为 `项目未实现` 或 `离线` 的 adapter/operation；如果矩阵已经是 `运行`、`原生`、`组合`、`REST兜底`，或已明确 `不支持` 且原因完整，应移出“项目未实现/未启用”段，只在 adapter 文档保留边界说明。同一 adapter 可以同时出现在“项目未实现”和“当前不支持”段，但必须对应不同 operation，例如 amend 离线、batch/OCO 不支持。
